## Intent

Replace the hard-coded `SECTIONS` list in [back-end/file_processing/build_azure_policies.py](back-end/file_processing/build_azure_policies.py)
so the generated `azure_policies.docx` mirrors the **new** Azure policy PDF the
developer attaches. The new document has **more than 15 sections** and different
titles. All downstream consumers (search index ingestion, categorisation rules,
test invariants) must be updated in the same change so the smoke test, the
build-index pipeline, and the search query layer continue to work.

## Hard structural contract — do not break

Two extractors depend on a **strict header convention**. Both live in the repo
and key off the same rule:

- [back-end/search/build_index.py](back-end/search/build_index.py) `policies_from_docx()` — line ~62, treats a paragraph as a section header **iff** all three are true:
  1. `line.isupper()`
  2. `"INTERNAL" not in line`
  3. `line[:1].isdigit()`
- [back-end/file_processing/parsing.py](back-end/file_processing/parsing.py) `extract_policies_docx()` — same rule.

**Therefore every header Claude emits MUST match this regex shape:**

```
^\d+\.\s+[A-Z][A-Z0-9 \-]*$
```

Examples that pass: `"1. IDENTITY AND ACCESS MANAGEMENT"`, `"23. AI AND AGENT WORKLOADS GOVERNANCE"`.
Examples that fail: `"1. Identity and Access Management"` (mixed case), `"Identity"` (no number), `"1.1 Sub-section"` (sub-numbering — flatten or skip).

If the new PDF uses sub-sections (e.g. `2.1`, `2.2`), **flatten them**: either
fold sub-section bodies into the parent section, or promote each leaf to a
top-level numbered section with a continuous counter. Pick one strategy and
apply it consistently — do not mix.

## Edge cases Claude must handle

1. **Sub-numbering in the new PDF** — flatten as above; never emit `1.1` headers.
2. **Title characters** — strip punctuation other than spaces and hyphens; uppercase
   the whole title; do not include trailing colons or em-dashes.
3. **Image-only or table-only sections** — extract any caption / surrounding
   prose; if a section is purely a table, render the table's intent as 1–2
   sentences of prose so the indexer has searchable text.
4. **Very short sections (< ~80 words)** — keep them; do not merge into a
   neighbour. The indexer chunks at 3500 chars so short sections are fine.
5. **Very long sections (> ~3500 chars)** — leave the body intact in the
   `SECTIONS` tuple. The current builder splits paragraphs at sentence
   boundaries (3 chunks). Preserve that behaviour; do not pre-chunk in the
   tuple.
6. **Duplicate or near-duplicate titles** — disambiguate by appending a
   distinguishing noun from the body (e.g. `"NETWORK SECURITY - INGRESS"` vs
   `"NETWORK SECURITY - EGRESS"`).
7. **New policy domains not covered by `categorize.py`** — add a new tuple to
   `_RULES` in [categorize.py](back-end/search/categorize.py) with
   discriminating keywords. Order matters — earlier rules win.
8. **Document title / preamble paragraphs** — keep them non-numeric and
   non-uppercase so the extractor ignores them, exactly like the existing
   builder does.

## Body text — verbatim copy from the new PDF

The developer has chosen **verbatim copy** for body content. Claude must:

- Copy each section's prose from the new PDF exactly, preserving wording.
- Strip page numbers, headers/footers, and OCR artefacts.
- Normalise whitespace (collapse runs of spaces, remove stray line breaks
  inside paragraphs) but do **not** rewrite sentences.
- If the PDF contains bullet lists, render them as comma-separated prose or as
  paragraph sentences — the DOCX builder currently emits paragraphs only and
  the indexer treats the body as a single chunked string.
- Replace the existing module docstring's claim that the content is
  "originally written" — the tool runs in the customer's own environment
  against their own policy document, so the new docstring should read:
  *"Verbatim copy of the customer-supplied policy document
  `<source-pdf-filename>`, rendered as DOCX for the ingestion pipeline."*

## Procedure Claude should follow

1. Read [back-end/file_processing/build_azure_policies.py](back-end/file_processing/build_azure_policies.py) to confirm the output shape and the `SECTIONS` schema (`list[tuple[str, str]]`).
2. Read [back-end/search/build_index.py](back-end/search/build_index.py) `policies_from_docx()` to re-confirm the header rule.
3. Read [back-end/search/categorize.py](back-end/search/categorize.py) `_RULES`.
4. Read [back-end/tests/test_policy_ingest.py](back-end/tests/test_policy_ingest.py) `test_policies_docx_has_fifteen_sections`.
5. Extract every section from the attached new PDF. Produce an in-memory list of `(header, body)` tuples obeying the structural contract above.
6. Rewrite `SECTIONS` in `build_azure_policies.py`. Update the module docstring's section-count phrasing (currently *"15 sections"*).
7. Update `_RULES` in `categorize.py` only for genuinely new domains.
8. Update the test: change `assert len(sections) == 15` to the new exact count, and update the docstring.
9. Run the smoke test locally: `pytest back-end/tests/test_policy_ingest.py::test_policies_docx_has_fifteen_sections -q` (rename the test function if the count changes, e.g. `test_policies_docx_has_N_sections`).
10. Regenerate the DOCX: `python back-end/file_processing/build_azure_policies.py`.
11. Report back: new section count, list of new categories added to `_RULES`, and any sections where flattening or disambiguation was applied.
